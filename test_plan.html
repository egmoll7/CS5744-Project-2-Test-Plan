Test Plan

Our integration plan and requirements for testing will be based on the access and usability between the different types of users. The application should have certain functions that not all users can access. It should also have the ability for users to make notes that Veterinarians can view, and no other user can edit the Veterinarians notes, but the individual Veterinarian. The application will allow many different types of users to interact with a database, each authorized user has their own level of access restrictions.

We believe that the best Plan Build and Test Plan for this system will be a Boehmâ€™s Spiral Model. The testing should be integrated at each prototype because the application is dealing with sensitive information that needs to be securely validated. The system is intended to be used where it is not essential for all of the intended end users to have access to all of the program operations and information. The system can be built for one set of users at a time. This will give the building team time to build the application for the Veterinarian user that does not have any restrictions. Then to create additional user restrictions as each prototype is developed.

This will ensure that the most important type of users have access to the full performance of the application. Performing this type of testing, the builders can ensure that they are testing the different users level access, from the most important, Veterinarians, to the last phase of users, like Groomers and Travel Agencies. Which is why we believe that the Spiral Model of testing is the best for this application. Each phase will allow for the builders to ensure that the application is meeting the objectives of the phases intended clients, addressing one clients risks at a time and developing one clients platform at a time.

Application Deployment & Client

Prototype Phases

The Test Procedure for this application is going to consist of 4 prototypes: building team will build the prototypes for each specific type of user, gather information of the users interaction and performance and business integration. The users will be tested in isolation to ensure they have the correct interaction with the application and the testing scenarios.

Prototype 1

For the first prototype, the builders should focus on the users with the least amount of application restrictions. So that the first set of test scenarios can focus on the performance of the application, minimizing the applications integrity risks. The test will be focus on User Interactions and Reusability.

Phase Build Test
1 Objective: Focus on the interaction the Veterinarian user type will have with the database. Testing that the application is useful to the intended user.
2 Risk: Builders risk at this point is to ensure that the information can be accessed and transferred easily between offices. Testing the ability of the application to be integrated across different platforms.
3 Develop: Creating a platform for the first intended users with no restrictions for the Veterinarian. Testing that the first prototype is running and all the basic functionality of the application is working properly.
4 Planning the next phase: Pet Owner. Testing that all the needs from the intended user are met. (UAT verification)

Prototype 2

In the second prototype, we are going to be testing the interaction between the Pet Owner and the application. This will include the first test scenarios of user restrictions focusing on performance. The test will focus on User Interaction and User Type Restrictions.

Phase Build Test
1 Objective: Create the first set of user restrictions for the Pet Owner and creating a Pet Owner Interaction Page. Testing that the application is useful to the Pet Owner and the application objectives meet the Pet Owner needs.
2 Risk: The Pet Owner has the ability to change any animal's medical record. Testing the restrictions are set properly.
3 Develop: The Pet Owners User Platform and Pet Owner restrictions to medical records. Testing the ability for different type of users. Prove that the user is a Veterinarian, giving to that user type the ability to change any medical records. Showing records to Pet Owners and giving them the ability to make separate notes.
4 Planning the next phase: Animal Shelters and Adoption Agencies. Testing that all the needs from the intended user are met. (UAT verification)

Prototype 3

In the third prototype, we are looking to test the availability of a platform for Animal Shelters and Adoption Agencies. In this prototype we are going to focus on the availability of any previous medical bills or previous Owners notes are available to future Owners and future Veterinarians. The test will focus on User Interaction and User Type Restrictions.

Phase Build Test
1 Objective: Create a second set of user restrictions for Animal Shelters and Adoption Agencies. Testing that the application is useful to the Animal Shelter and Adoption Agencies and the application objectives meet the Animal Shelters and Adoption Agencies needs.
2 Risk: The ability for Adoption Agencies to change previous Owners or Veterinarians notes. Testing the restrictions are set properly.
3 Develop: The platform for Animal Shelter and Adoption Agencies. Testing that the existing Owners and Veterinarian notes can't be changed and giving them the ability to make separate notes.
4 Planning the next phase: Groomers and Travel Agencies Testing that all the needs from the intended user are met. (UAT verification)

Prototype 4

In the fourth and final prototype, we are not only going to do testing for restrictions on editing information but on information access. The focus on this testing scenario is to ensure that the Travel Agencies only see the pertinent medical information and Groomers can only see the pertinent Pet Owner notes information.

Phase Build Test
1 Objective: Final set of restrictions, not all the notes and medical bills will be shown to the Groomers and Travel Agencies user types. Only the pertinent information will be available to the appropiate user types. Testing that the intended user objectives can be met while the user types have viewing access restrictions.
2 Risk: The ability for Groomers and Travel Agencies to access Owners notes and medical record information that they do not need. Testing the restrictions are set properly.
3 Develop: The final version of the application. Testing that all the access and editing information is fully developed and that the Groomers and Travel Agencies can see and edit what was previously set as objectives. (Final UAT verification)
